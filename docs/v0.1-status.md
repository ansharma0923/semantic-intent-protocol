# SIP v0.1 Status

**Status:** Stable reference implementation
**Version:** 0.1
**Date:** 2026

---

## Overview

SIP v0.1 represents the first stable version of the Semantic Intent Protocol specification and reference implementation.

The protocol defines a deterministic control plane for AI agents and software systems to express intent, negotiate capabilities, enforce policy, and produce execution plans before interacting with external execution systems. This release establishes the foundational architecture and security model of SIP and serves as the canonical reference for implementers.

---

## Stable Components

The following components are considered stable in SIP v0.1:

### IntentEnvelope Schema

The `IntentEnvelope` is the core protocol object. It carries the actor identity, trust context, intent declaration, constraints, provenance block, and optional extension fields. The schema is fully specified and all fields are validated deterministically at submission time.

### CapabilityDescriptor Model

The `CapabilityDescriptor` describes a capability that can satisfy an intent. Each descriptor declares supported intent domains, operation classes, required scopes, trust level requirements, risk level, and supported execution bindings. Capability descriptors are registered in the capability registry and used during negotiation.

### NegotiationResult and Deterministic Capability Matching

The negotiation engine matches a submitted `IntentEnvelope` against all registered capabilities and produces a ranked `NegotiationResult`. Matching is deterministic: given the same envelope and registry state, the broker always selects the same capability.

### Policy Engine and Authorization Model

The policy engine evaluates every submitted intent against a rule-based authorization model. It performs scope checking, trust level enforcement, risk-based approval routing, data sensitivity gating, and delegation chain validation. All policy decisions are deterministic and produce a clear allow, deny, or require-approval outcome.

### ExecutionPlan Generation

Once a capability is selected and policy is satisfied, the broker produces a fully specified `ExecutionPlan`. The plan includes the target capability, the chosen execution binding, translated request payload, and all parameters required for execution. The broker does not perform execution itself.

### AuditRecord Logging

Every intent processed by the broker produces an `AuditRecord` capturing the actor identity, intent details, selected capability, policy decision, and outcome. Records include originator and delegation chain fields to support post-hoc provenance analysis. Audit records are immutable and must be appended to a durable log in production deployments.

### Intent Provenance and Intent Laundering Protections

The optional `ProvenanceBlock` on each `IntentEnvelope` records the originator separately from the submitting actor. The policy engine enforces authority monotonicity: delegated authority can never exceed the originator's authority. This prevents intent laundering attacks in which a trusted agent mediates a low-privilege request into a high-privilege structured intent.

### Broker HTTP API

The SIP broker exposes a FastAPI-based HTTP interface as a transport surface for the protocol. Key endpoints include:

- `POST /sip/intents` – submit an `IntentEnvelope` and receive a structured broker response.
- `GET /capabilities` – list registered capabilities.
- `GET /healthz` – liveness and readiness check.

The HTTP API is one transport surface for SIP and is not the protocol definition itself.

### Capability Registry (In-Memory and Persistent JSON)

The capability registry stores `CapabilityDescriptor` objects and makes them available for negotiation and discovery. SIP v0.1 supports both an in-memory registry and a file-backed JSON registry (`JsonFileCapabilityStore`). The persistent store saves capabilities to `data/capabilities.json` by default and reloads them on startup. The storage path is configurable via `SIP_CAPABILITIES_FILE`.

### External Identity Integration

SIP v0.1 supports mapping pre-authenticated identity claims from trusted HTTP headers (`X-Actor-Id`, `X-Actor-Type`, `X-Actor-Name`, `X-Trust-Level`, `X-Scopes`) into the SIP actor and trust context. Authentication remains external to SIP; the broker only maps pre-verified claims provided by a trusted gateway or service mesh. This feature is opt-in and is enabled via `SIP_TRUSTED_IDENTITY_HEADERS=true`.

### Protocol Extension Mechanism

All core protocol objects (`IntentEnvelope`, `CapabilityDescriptor`, `NegotiationResult`, `ExecutionPlan`, `AuditRecord`) support an optional `extensions` field for carrying vendor-specific or application-specific metadata. Extension keys must use the `x_<name>` or `<vendor>.<name>` format. Unknown extensions are preserved and never affect core protocol processing. See `sip/extensions.py` for validation rules.

### Capability Discovery API

The capability discovery API provides structured endpoints for querying the registry:

- `GET /sip/capabilities` – list all registered capabilities as full descriptors.
- `GET /sip/capabilities/{id}` – retrieve a capability by its unique identifier.
- `POST /sip/capabilities/discover` – submit a `DiscoveryRequest` and receive ranked capability candidates.

Discovery requests support filtering by intent name, intent domain, operation class, preferred bindings, and trust level.

### Distributed Broker Support

The `DiscoveryService` supports optional peer broker federation. When a `FederationConfig` is provided, discovery requests are forwarded to configured peer brokers via HTTP and results are aggregated deterministically. Local capabilities are preferred by default. Peer failures are handled gracefully in soft mode or cause fast-fail in strict mode.

### Federation Model

SIP v0.1 includes a trust-aware multi-broker federation model built on `FederationConfig` and `FederatedPeer`. Three peer trust levels control capability visibility:

- `DISCOVERY` – remote capabilities appear in discovery results but are not eligible for routing.
- `ROUTING` – remote capabilities may be included in execution plans.
- `FULL` – peer is fully trusted for discovery, routing, and execution delegation metadata.

Remote capabilities are tagged with source broker provenance. The local broker is always the final policy authority regardless of peer trust level.

## Execution Bindings

SIP v0.1 supports the following deterministic execution bindings:

- **REST** – HTTP/REST adapter producing request translation payloads for REST endpoints.
- **gRPC** – gRPC adapter producing method and message translation payloads.
- **MCP** – Model Context Protocol adapter for AI model context operations.
- **A2A** – Agent-to-Agent adapter for inter-agent delegation and execution.
- **RAG** – Retrieval-Augmented Generation adapter for retrieval system integration.

SIP produces execution plans and translation payloads for these bindings. The broker does not perform execution itself; execution is delegated to the appropriate adapter or execution system after the plan is produced.

## Security Model

SIP v0.1 implements the following security guarantees:

- **Trust levels** – Every actor carries a declared trust level (`public`, `internal`, `privileged`, `admin`). Operations are gated on the actor's trust level, and declared trust levels may not exceed the actor's assigned level.

- **Scope-based authorization** – Each capability declares the scopes required to invoke it. The policy engine denies requests with insufficient scopes before any other evaluation occurs.

- **Risk-based approval requirements** – High-risk operations require explicit human approval before execution proceeds. Critical-risk operations require approval regardless of operation class.

- **Delegation chain limits** – Delegation depth is bounded at 5 hops. Chains exceeding this limit are denied to prevent infinite re-delegation cycles.

- **Provenance-aware authorization** – The policy engine evaluates both the submitting actor and the originator when a provenance block is present. The effective authority is the lower of the two.

- **Intent laundering protections** – Authority monotonicity is enforced across all delegation steps. A delegated intent cannot gain scopes or trust level exceeding those held by the originator.

- **Immutable audit records** – Every processed intent produces an audit record capturing the full actor identity, intent details, policy decision, and delegation chain. Records must be written to an append-only log in production deployments.

For full details, see [security-model.md](security-model.md).

## Determinism Guarantees

The SIP protocol pipeline is deterministic.

For the same `IntentEnvelope` and registry state, the broker will always produce the same `NegotiationResult` and `ExecutionPlan`. Capability matching, policy evaluation, and execution plan generation are entirely rule-based with no probabilistic or heuristic components.

AI agents or LLMs may generate `IntentEnvelope` objects outside the protocol layer. SIP itself does not perform LLM inference. The separation ensures that non-deterministic AI reasoning remains outside the security boundary, while SIP enforces deterministic, auditable control over what is authorized and planned for execution.

## Current Limitations

The following are intentional design boundaries of SIP v0.1:

- **Static federation trust configuration** – Federation peer trust levels are declared in configuration at startup. Dynamic peer trust negotiation is not supported in v0.1.

- **JSON-based registry persistence** – The persistent capability registry uses a flat JSON file. Concurrent multi-process writes and large-scale registry deployments are outside the scope of v0.1.

- **No global capability discovery network** – Capability discovery is scoped to a single broker or a statically configured set of federated peers. There is no global discovery registry or namespace in v0.1.

- **No distributed consensus between brokers** – Federated brokers do not synchronize state or perform distributed coordination. Each broker maintains its own authoritative registry. Peer discovery results are aggregated locally but not replicated.

These boundaries reflect the scope of the v0.1 specification and are not deficiencies in the implementation.

## Future Evolution

Future SIP versions (v0.2 and beyond) may evolve the following areas:

- **Federation governance** – Dynamic peer enrollment, trust negotiation, and federated policy propagation.
- **Distributed registry synchronization** – Capability state replication and eventual consistency across broker nodes.
- **Advanced identity integration** – Deeper integration with identity providers and verifiable credentials.
- **Capability schema versioning** – Explicit schema versioning for `CapabilityDescriptor` to support capability lifecycle management.
- **Ecosystem interoperability** – Alignment with emerging standards and interoperability with other AI agent frameworks and protocol ecosystems.

No detailed roadmap is defined for these areas. They represent directions for community exploration beyond the v0.1 baseline.

---

## Compatibility Policy

SIP v0.1 aims to maintain backward compatibility within the 0.x minor series where possible.

- **Extensions provide forward compatibility** – The `extensions` field on all core protocol objects allows new metadata to be carried without modifying the core schema.
- **Unknown extension fields must be ignored** – Implementations must preserve unknown extension fields and must not reject objects containing unfamiliar extension keys.
- **Breaking protocol changes require a version increment** – Any change to required fields, validation rules, or negotiation semantics that would break existing conforming implementations will be introduced under a new protocol version.

---

## Summary

SIP v0.1 establishes the foundational protocol, architecture, and security model required for safe semantic interoperability between AI agents and software systems. The stable components documented here provide a complete reference implementation covering intent expression, capability negotiation, policy enforcement, execution planning, auditing, federation, and extensibility. This release is the intended baseline for implementers building systems that integrate with or extend the Semantic Intent Protocol.
