# SIP v0.1.1 Release Notes

**Release date:** 2026-03-12
**Protocol version:** SIP v0.1
**Status:** Ecosystem foundation release

---

## Overview

SIP v0.1.1 is the first numbered release of the **Semantic Intent Protocol** reference implementation.
It establishes the complete v0.1 protocol baseline, including Python and Go SDKs, a runnable broker HTTP API, capability discovery, federation, and a full interoperability test suite.

This release is intended for early adopters and integrators who want to build on SIP before the public protocol specification is finalized.

---

## Highlights

### Deterministic Semantic Intent Protocol

SIP defines a structured intent layer above existing execution protocols (REST, gRPC, MCP, A2A, RAG).
Every `IntentEnvelope` carries:

- **actor** — who is making the request (ID, type, trust level, scopes)
- **intent** — what they want (name, domain, operation class, parameters)
- **desired outcome** — success criteria and output format
- **constraints** — deadline, idempotency, latency budget
- **provenance** — originator and delegation chain
- **protocol bindings** — preferred execution protocols

### Python SDK

The `sip.sdk` module provides a clean, stable public API for Python applications:

- `build_actor()`, `build_intent_envelope()`, `build_provenance()` — builder helpers
- `BrokerClient` — HTTP client for submitting intents
- `CapabilityDiscoveryClient` — HTTP client for capability discovery
- `validate_envelope()` — schema and semantic validation
- `to_json()`, `to_dict()`, `parse_intent_envelope()` — serialization
- `SIPError`, `SIPHTTPError`, `SIPValidationError` — typed exceptions

### Go SDK

`sdk/go/sip` provides full SIP v0.1 type coverage in Go:

- Complete struct definitions for all protocol objects
- Constructor helpers (`NewIntentEnvelope`, `NewActorDescriptor`, etc.)
- `BrokerClient` — HTTP client matching the Python client's capabilities
- JSON serialization compatible with the Python reference implementation
- Protocol vector test suite for cross-SDK compatibility

### Broker HTTP API

The SIP broker is a FastAPI application (`sip.broker.service:app`) that exposes:

| Endpoint | Description |
|----------|-------------|
| `GET /healthz` | Broker health and capability count |
| `POST /sip/intents` | Submit an `IntentEnvelope` for processing |
| `GET /sip/capabilities` | List all registered capabilities |
| `GET /sip/capabilities/{id}` | Fetch a capability by ID |
| `POST /sip/capabilities/discover` | Semantic capability discovery query |

### Capability Discovery API

The discovery API supports semantic queries by intent name, domain, operation class, and binding type.
Results are ranked by match score and include federated candidates from peer brokers.

### Persistent Registry

Capabilities can be persisted to a JSON file using `JsonFileCapabilityStore`.
The broker reloads capabilities from the file on startup.

### Distributed Brokers

Each `BrokerService` instance is fully independent and can operate standalone or as part of a federation.

### Federation Model

`FederationConfig` and `FederatedPeer` allow a broker to query peer brokers during discovery.
Trust levels control whether remote capabilities are eligible for routing:

| `PeerTrustLevel` | Included in Discovery | Eligible for Routing |
|-----------------|----------------------|----------------------|
| `DISCOVERY`      | ✓                    | ✗                    |
| `ROUTING`        | ✓                    | ✓                    |
| `FULL`           | ✓                    | ✓                    |

### Intent Provenance Protections

`ProvenanceBlock` records the originator, submitted-by actor, delegation chain, delegation purpose,
and the authority scope granted. This information is immutable in the audit record and cannot be
overwritten by intermediate brokers or adapters.

### Protocol Extension Support

`IntentEnvelope.extensions` and `CapabilityDescriptor.extensions` accept arbitrary vendor or
application metadata under namespaced keys (e.g., `"x-acme.priority_queue": true`).
Extension keys starting with `x-sip.` are reserved for official protocol use.

### Interoperability Test Suite

`tests/interoperability/` contains multi-broker federation tests.
`tests/protocol_vectors/` validates all canonical JSON fixtures against the Python implementation.

### Protocol Vectors

Six canonical JSON fixtures in `protocol-vectors/` define the normative wire format for:

- `intent-envelope-basic.json`
- `intent-envelope-with-provenance.json`
- `capability-descriptor-basic.json`
- `negotiation-result-basic.json`
- `execution-plan-basic.json`
- `audit-record-basic.json`

## Architecture Overview

SIP uses a **control plane / data plane separation** model:

```
┌────────────────────────────────────────────────┐
│                  SIP Control Plane              │
│                                                 │
│  Actor → IntentEnvelope → BrokerService         │
│                 │                               │
│         ┌───────▼──────────┐                    │
│         │  CapabilityMatcher│ (scoring)          │
│         └───────┬──────────┘                    │
│                 │ NegotiationResult              │
│         ┌───────▼──────────┐                    │
│         │  PolicyEngine    │ (scopes, risk, approvals) │
│         └───────┬──────────┘                    │
│                 │ PolicyDecision                 │
│         ┌───────▼──────────┐                    │
│         │  ExecutionPlanner│                    │
│         └───────┬──────────┘                    │
│                 │ ExecutionPlan + AuditRecord    │
└─────────────────┼──────────────────────────────-┘
                  │
┌─────────────────▼──────────────────────────────┐
│                  Adapter Layer                  │
│  REST  |  gRPC  |  MCP  |  A2A  |  RAG         │
└────────────────────────────────────────────────┘
```

The broker's **DiscoveryService** can aggregate candidates from peer brokers via the federation model,
but all policy evaluation happens locally on the receiving broker.

## Security

### Trust Levels

Every actor declares one of four trust levels: `public`, `internal`, `privileged`, `admin`.
Capabilities declare a `minimum_trust_tier`; requests from actors below that tier are denied.

### Scopes

Every actor carries a set of scopes (e.g., `sip:knowledge:read`).
Every capability declares `required_scopes`; the intersection must satisfy the requirement.

### Policy Evaluation

The `PolicyEngine` evaluates every intent against:

1. Scope sufficiency
2. Trust tier
3. Risk level (capabilities are tagged `low`, `medium`, `high`, `critical`)
4. Data sensitivity
5. Delegation chain integrity
6. Approval requirements (configurable per risk level)

### Provenance Protections

`ProvenanceBlock` fields are set by the submitting system and immutably recorded in every `AuditRecord`.
Peer brokers cannot modify the originator or delegation chain fields during federation discovery.

## Compatibility

SIP v0.1.1 establishes the **SIP v0.1 baseline**.

- All v0.1 protocol vectors are normative and will remain valid in future v0.1.x releases.
- The Python SDK `sip.sdk` public API is stable for v0.1.x.
- The Go SDK `sdk/go/sip` package API is stable for v0.1.x.
- The broker HTTP API endpoints and response shapes are stable for v0.1.x.

Breaking changes require a new minor version (v0.2+).
